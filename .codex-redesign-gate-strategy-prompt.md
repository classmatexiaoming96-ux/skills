## 你要做的事：重新设计策略

### 核心问题重新定义

效铭的原话：这种页面就是反面典型啊 怎么能顺利通过呢。

**真正的反面典型信号**：
- 大量 code path:NN 引用前后 200 字符内**没有中文/英文讲解**
- 整页多个 H3 段落都只引用行号，不解释原理
- 源码索引段本身可以是合规的（每个文件 1 个 :1 是合理的入口索引），但**主页面不能所有行号都靠这个段落凑**

### 新策略设计原则

1. **行号必须配讲解**：每个 code path:NN 引用前后 200 字符内有 ≥30 字符的中文/英文讲解（保留）
2. **讲解密度是硬指标**：≥70% 的 code path:NN 必须有讲解（**从 warning 升到 ERROR**）
3. **行号堆砌段的定义更严格**：
   - 连续 8 个 code path:NN 分隔符就报错（**降低阈值**，原 15）
   - **去掉源码索引/源文件地图白名单**（这是上一轮的漏洞）
   - 例外：每个文件路径首次出现用 :1 引用是允许的，但只能出现 1 次
4. **code_wrap 阈值提上去**：主页面 ≥ 6，子页面 ≥ 3（**恢复原值**）
5. **讲解段落数是新增硬指标**：
   - 统计页面里有几个**独立讲解段落**（≥100 字符中文/英文 + 含 path:NN 引用）
   - 主页面 ≥ 6 个独立讲解段落
   - 子页面 ≥ 3 个
   - 这条直接抓是否真的讲了原理
6. **index.html 缺卡片回到 ERROR**：用户看不到的页面不算交付
7. **可选：行号真实性检查**：加 --sources 参数时，验证每个 code path:NN 引用的行号确实存在于源码文件（防止 agent 编造行号）
